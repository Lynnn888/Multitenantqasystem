# 工程决策文档

## 系统架构
Client → FastAPI接口层
       → 多租户隔离层（独立目录+独立向量索引）
       → 文件摄取Pipeline（解析→分块→向量化→存储）
       → 混合检索（向量检索 + BM25 → RRF融合）
       → LLM生成 + SSE流式输出
       → 配额统计与限流

## 关键技术决策

### 1. Embedding 与向量库选择
选用 sentence-transformers + FAISS-CPU
理由：
- 轻量无需额外部署，适合限时快速开发
- 本地可直接运行，不依赖外部向量数据库
- 满足作业演示级检索需求

未选用：Pinecone、Chroma、PGVector，因为部署耗时超出3小时时限。

### 2. Chunk 分块策略
参数：chunk_size=512，chunk_overlap=64
测试过 256 / 512 / 1024：
- 256：检索准但语义断裂
- 1024：语义完整但检索精度低
- 512：平衡语义完整性与检索准确度，最终采用

### 3. 混合检索方案
采用：向量相似度检索 + BM25关键词检索，RRF融合排序
优势：兼顾语义匹配与关键词精准匹配，比单路检索召回更好。

## 矛盾约束取舍

### 1. 混合检索延迟 vs 流式首token速度
做法：两路检索并行执行，提前预加载模型，减少初始化耗时，保证首token响应速度。

### 2. 多租户隔离安全
采用物理隔离（独立文件夹、独立向量索引）+ 代码租户ID双重校验，杜绝跨租户泄露。

### 3. 块大小权衡
折中选择512，兼顾语义完整和检索精度。

## 已知待优化问题
1. 向量索引为文件存储，高并发性能一般
2. 配额为硬编码，未做数据库持久化
3. RRF权重固定，未做动态调参
4. 异常文件容错处理有待完善
